# distance.py 設計改善 提案（診断と方針）

> 作成日: 2026-06-15
> 対象: `src/distance.py`（新規トピック抽出）周辺
> ステータス: 提案（未実装）

## 背景・要件

新規トピックに求められる要件:

- **R1（包含）**: これまでの要約記事と比べて新規の内容を、記事冒頭で簡潔に確認できること
- **R2（排除）**: 直近のスレで何度も話題になっている要素を排除すること
  - 実例: 過去にAnimaというモデルが流行していたとき、新規トピックが毎回Animaの話題のみになってしまい、新規トピックの役割を果たせなかった

---

## 現行設計の何が要件に合っていないか（診断）

現行 `distance.py` の判定は **「全履歴の中で最も近い1チャンクへの距離 > 閾値なら新規」** である。これは本質的に **「言い回しがほぼ重複していないか（near-duplicate 判定）」** を測っているだけで、要件が求める **「トピックとして直近で繰り返されているか（recurrence）」** を測っていない。ここに要件とのズレがある。

### Animaが毎回新規になった理由

Anima は毎スレ話題になるが、各スレで言い回し・細部が少しずつ変わる。すると:

- 今スレのAnimaチャンクは、過去のどのAnimaチャンクとも「完全一致」はしない → 距離 > 0.07 → **毎回「新規」と誤判定**
- しかもAnimaは言及量が多くチャンク数も多いので、**新規欄がAnimaで埋め尽くされる**

つまり現行metricは「定常的に議論され続けている人気トピック」と「本当に新しいトピック」を区別できない。near-duplicate しか見ていないので、**リワードされ続ける人気トピックほどすり抜けて新規扱いになる**という逆説が起きる。

### 要件の再定式化

求めているのは1つのmetricではなく、**2軸**である:

- **新規性**: 直近の議論に登場していない
- **非飽和性**: 直近の複数スレで繰り返されていない（＝Animaのような背景ノイズでない）

どちらも「**全履歴**」ではなく「**直近の窓（recent window）**」に対して、「**完全一致**」ではなく「**トピックとして近いか（広めの半径）**」で測るべきである。

## 提案：直近窓 × トピック再出現スコアリング（Recency-windowed recurrence）

### 中核アイデア

各候補チャンクについて「全履歴の最近傍距離」ではなく、**「直近Nスレのうち、意味的に近いチャンクが登場したスレ数（= recurrence / 文書頻度）」** を数える。

```
N    : 直近窓のスレ数（例 6）
r    : トピック一致半径（例 0.15。現行0.07より「広め」にして言い換えを同一トピックとして捕捉）
SAT  : 飽和判定（例 直近Nスレの50%以上に出る → 飽和とみなす）

recent_threads = 「現在より前で実在する直近Nスレ番号」  # 算術範囲でなく実在集合の上位N（欠番に強い）

for chunk in current_chunks:
    近傍 = collection.query(chunk, n_results=多め,
                            where={"thread_num": {"$in": recent_threads}})
    hit_threads = { 距離<=r の近傍が属する thread_num の集合 }
    recurrence = len(hit_threads)

    if recurrence >= SAT*N:   → 飽和（Animaなど）→ 除外
    elif recurrence == 0:     → 完全新規 → 採用（高優先）
    else (1..少数):           → 直近で出始めた新興 → 採用（通常優先）
```

### なぜこれで両要件を満たすか

- **Anima問題が解ける**: Animaは直近ほぼ全スレに登場 → recurrence高 → 飽和除外。言い回しが毎回変わっても、半径rを広めに取るので「同じトピック」として確実にヒットする（現行が0.07で取りこぼしていた所）。
- **新規を冒頭で簡潔に**: recurrence=0/低のものだけ残るので、digestは本当に新しい話題に絞られる。
- **副次効果（自己マッチ撲滅）**: 窓は「現在より前のスレ」だけなので、構造的に自スレを含まない。→ これまでの「再実行前にDBから当該スレを削除」という運用前提が **不要** になる（store_to_chromaはid upsertで冪等）。

### 概念的裏付け

これは実質「**トピックレベルのTF-IDF**」である。recurrence = そのトピックの直近窓での文書頻度（DF）であり、高DFトピックはストップワードのように抑制される、という標準的な発想を ChromaDB 上でオンライン計算しているだけ。

## mod.md の精度強化（任意・推奨）：エンティティ再出現

mod.md は各チャンクに `models`（例 `"Anima, Anim, ..."`）と `model`（見出し）が入っている。これを使えば、**モデル名の直近窓での出現スレ数**を直接カウントできる:

- あるモデルが直近Nスレの大半に出る → 飽和 → 除外（Animaにドンピシャ）
- 直近窓に **初登場** するモデル → 強い新規シグナル → 優先採用

意味的recurrence（全ファイル共通）に加え、mod.md だけはこのエンティティ単位カウントを併用すると、ほぼ決定論的かつ高精度に「新モデルだけ」を抽出できる。

## LLM段の役割（改訂2の方針を維持）

選別は上記の決定論スコアリングが担当し、LLMは **採用済みチャンクの凝縮のみ**。recurrenceスコアをLLMに渡せば「**新登場**」「直近で **話題化**」のように温度感を付けた表現も可能。異常時は決定論整形へフォールバック（現状通り）。

## パラメータと移行

- env で調整可能に（既存prefilterに倣う）: `NOVELTY_WINDOW`(N)、`NOVELTY_RADIUS`(r)、`NOVELTY_SATURATION`(SAT)。ファイル別に既定値（sum/mod/tools）。
- thread_num はゼロ埋め文字列なので、**実在thread_numを集めて降順上位N** を窓にする（欠番・非連番に強い）。
- 初期スレ（窓に十分な過去が無い）は自動的に「ほぼ全部新規」に縮退（許容）。
- 後方互換: env未設定時は現行の単純距離判定にフォールバックする実装にすれば、安全に段階導入できる。
