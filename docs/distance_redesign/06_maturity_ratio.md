## 0. これまでの検証の流れ（要約）

新規トピック抽出は「R1: 過去記事と比べた新規内容を冒頭に簡潔提示」「R2: 直近で繰り返される定番（Anima等）を排除」を満たす必要がある。設計を以下の順で検証・改訂してきた。

| 段階 | 内容 | 結論 | 参照 |
|---|---|---|---|
| 旧実装 | 全履歴の最近傍距離 > 閾値 → 新規 | near-duplicate しか見ておらず、Anima がすり抜け新規欄を埋める | — |
| 提案1 | 直近窓×トピック再出現スコア | 新興ホット話題(Ideogram4等)が定番(Anima)と同程度に再出現し**分離不能** | 01,02,03,04 |
| 検証 | 初出年齢 | seedance/gemma4/hidream など採用したい話題も初出が古く不可 | 04 |
| 検証 | 意味ベクトルの全履歴DF | 半径0.15で全件高DF、0.10は逐語繰り返し検出のみで脆い | 04 |
| **採用** | **エンティティ（モデル名/ツール名）の全履歴DF（topic-IDF）** | 定番(高DF)と新興(低DF)が**クリーンに分離** | 04 |
| 実装検証 | sum/tools のキーワードDF | `keywords` が GiNZA の任意5語で非決定的 → **無効** | 05 |
| 最終 | sum→`models`DF / tools→`tools`DF へ切替 | 全ファイルでエンティティDFに統一、機能を確認 | 05 |

**到達点**: 「あるトピックが過去スレで何割のスレに登場したか（global document frequency, DF）」を、curated 辞書（model_checked.csv / tool_checked.csv）由来のエンティティで測り、**DFが高い＝長年の定番＝抑制、DFが低い＝新興/新規＝採用** とする（古典的な IDF / ストップワード除去と同じ発想）。

---

## 1. MATURITY_RATIO の概要

`MATURITY_RATIO` は、**「あるチャンクを『定番（成熟）』とみなして新規トピックから除外する境界値」** である。

- 各エンティティ（モデル名/ツール名）の **DF比率** = 「そのエンティティが登場した過去スレ数 ÷ 過去スレ総数」。
  - 例: Anima は過去123スレ中61スレに登場 → 比率 0.49。Ideogram は4スレ → 0.03。
- チャンクの **成熟度スコア** = そのチャンクに含まれるエンティティのDF比率の**平均**（ストップワード除外後）。
- 成熟度スコア **≧ MATURITY_RATIO** なら「定番」と判定し**除外**。未満なら**採用**。

直感的には「**MATURITY_RATIO = 全スレの何割以上に出ているエンティティ群で構成されたチャンクを“もう定番”として捨てるか**」のダイヤルである。

- 低くする（例 0.3）→ より多くを定番扱い → **除外が増え、採用が絞られる**（厳しい）。
- 高くする（例 0.5）→ よほど偏在するエンティティだけ定番扱い → **採用が増える**（緩い）。

## 2. 詳細説明（判定アルゴリズム）

`novelty_filter.py` の判定は次の通り（`is_mature` / `build_df_ratio`）。

### 2.1 DF比率マップの構築（`build_df_ratio`）
- 現スレより前の**実在スレ番号**のみを母集団にする（`thread_num < current`）。欠番・自己マッチに強い。
- 各過去レコードの対象フィールド（mod/sum=`models`, tools=`tools`）をカンマ分割し、
  `エンティティ(lower) → 登場スレ番号の集合` を集計。
- 比率 = `len(登場スレ集合) / 過去スレ総数`。

### 2.2 チャンクのトークン抽出（`chunk_tokens`）
- 現チャンク本文に `extract_metadata.extract_metadata` を適用し、curated 辞書に一致する
  モデル名/ツール名を取得（格納時と同じ決定論的部分一致）。

### 2.3 成熟判定（`is_mature`）
```
ratios = [DF比率(t) for t in tokens]
ratios = [r for r in ratios if r <= STOPWORD_CEIL]   # ① ストップワード除外
if not ratios:            # ② 認識トークン無し
    return False          #    → 未知＝新出の可能性として「採用」
score = mean(ratios)      # ③ 平均集約
return score >= MATURITY_RATIO   # ④ しきい値判定（True=定番=除外）
```

各ステップの意図:
- **① STOPWORD_CEIL（既定0.85）**: 比率が極端に高い偏在トークンを除外。`検索文書:` 接頭辞由来の「文書/検索」等のノイズや、ほぼ全スレ常在の汎用語が平均を不当に押し上げて新規チャンクを誤抑制するのを防ぐ。
- **② 空トークン=採用**: curated 辞書に無い新エンティティ（例 seedance, Fable）は認識されず空になる。これは「未知＝新出の可能性が高い」とみなし**採用**する（安全側）。
- **③ 平均集約**: 「最小値」だと定番チャンクが1つの中位エンティティを含むだけで生き残ってしまう（旧検証で確認）。平均は「チャンク全体としてどれだけ定番寄りか」を表し、boilerplate（定番だらけ）を確実に捉える。
- **④ しきい値**: ここで `MATURITY_RATIO` が効く。

### 2.4 関連パラメータ（環境変数）
| 変数 | 既定 | 説明 |
|---|---|---|
| `NOVELTY_MATURITY_MOD` | 0.40 | mod.md のしきい値 |
| `NOVELTY_MATURITY_SUM` | 0.40 | sum.md のしきい値 |
| `NOVELTY_MATURITY_TOOLS` | 0.40 | tools.md のしきい値 |
| `NOVELTY_STOPWORD_CEIL` | 0.85 | これ超のDF比率トークンは成熟度計算から除外 |

## 3. 調整パターンの例

### 3.1 しきい値感度（659・dry-run・採用/除外）

| file | チャンク総数 | m=0.30 | m=0.35 | m=0.40(既定) | m=0.45 | m=0.50 |
|---|---|---|---|---|---|---|
| sum.md | 18 | 13/5 | 13/5 | 13/5 | 13/5 | 16/2 |
| mod.md | 11 | 6/5 | 9/2 | 9/2 | 9/2 | 9/2 |
| tools.md | 17 | 12/5 | 13/4 | 14/3 | 14/3 | 14/3 |

読み取り:
- **mod.md**: 0.30 は厳しすぎ（中位DFのモデルまで除外し5件カット）。0.35〜0.50 は安定して 9採用/2除外（Anima定番recap＋PLaMo誤抑制の2件）。**既定0.40は妥当**。
- **sum.md**: 0.30〜0.45 で安定 13/5、0.50 で緩くなり 16/2。boilerplate（スレッド概要）等をしっかり落としたいなら 0.40前後、落としすぎが気になるなら 0.50。
- **tools.md**: 0.40 で 14/3。Forge系（準定番）を残したいなら 0.45〜0.50。

### 3.2 運用パターン例

| ねらい | 設定例 | 効果 |
|---|---|---|
| **バランス（推奨初期値）** | 全ファイル 0.40 | 明確な定番（Anima/illustrious/FLUX等）を抑制しつつ新規を保持 |
| **積極的に絞る（冒頭を短く）** | mod/sum/tools = 0.30〜0.35 | 中位DFのトピックも除外。新規欄が短くなるが誤抑制（新興の取りこぼし）増 |
| **保守的に残す（取りこぼし回避）** | 0.45〜0.50 | 偏在エンティティだけ抑制。boilerplate が残りやすい |
| **ファイル別最適化** | mod=0.40, sum=0.40, tools=0.45 | tools の準定番(Forge)を残しつつ mod/sum は標準 |
| **ストップワードを強める** | STOPWORD_CEIL=0.75 | 0.75超の偏在語を無視。汎用語ノイズが強い場合に誤抑制を低減 |

### 3.3 調整時の着眼点
- **誤抑制が気になる**（採用したい新興が消える）→ MATURITY_RATIO を上げる、または該当エンティティが curated CSV に無いか確認（無ければ追記で「空＝採用」に戻る）。
- **boilerplate が残る**（毎スレ同じ概要が新規欄に出る）→ MATURITY_RATIO を下げる。
- **汎用語で誤抑制**（"検索/文書/モデル"等が効きすぎ）→ STOPWORD_CEIL を下げる。

## 4. 「再燃トピック」の論点（後日検討用）

global-DF は「累積でどれだけ定番か」を測るため、**長期休眠後に再注目された旧トピック**を「準定番」として抑制しがちという既知の弱点がある。

実例（659）:
| トピック | 全履歴DF比率 | 直近の登場 | 現状の扱い(m=0.4) |
|---|---|---|---|
| HiDream-E1/O1 | 0.32 | 長期休眠後 659 で再登場 | 採用（0.32<0.4）だが境界的 |
| Forge Neo | Forge=0.88 | 「位置づけ変化」で再注目 | 抑制されやすい（準定番扱い） |
| PLaMo/LLM-jp | 実質0（CSV未登録） | 659 新登場 | 「Anima構文」併記で誤抑制 |

### 検討すべき対応オプション（実装は未着手・要方針決定）
1. **何もしない（純global-DF）**: シンプル。再燃は取りこぼすが定番抑制は安定。
2. **直近不在ボーナス**: 「直近Nスレに不在 → 今スレで再登場」したエンティティを新規としてブースト（DFが高くても採用）。HiDream/Forge の再燃を拾える。
3. **初出年齢の併用**: 初出が最近（直近Kスレ以内）なら無条件採用。新登場を確実に拾うが、seedance のような「初出は古いが今回新展開」は拾えない。
4. **curated CSV の継続メンテ**: 新モデル/ツール（PLaMo, seedance, Fable 等）を逐次追記。誤抑制（定番名の併記による巻き込み）と取りこぼしの両方を実務的に低減。最も即効性が高い。
5. **エンティティ抽出の高度化**: 未知語NER等で「辞書に無い新エンティティ」を新規シグナルとして検出。効果大だが実装コスト大。

### 推奨の検討順
- まず **(4) CSVメンテ + 既定0.40** で運用し、実出力を観察。
- 「旧くからあるが再注目」を積極的に拾いたい要件が強ければ **(2) 直近不在ボーナス** を追加検討（global-DF と直近シグナルの二軸化）。

## 4-A. 追記（2026-06-17 検討結果）

### 4-A.1 「直近不在ボーナス」は現構成では不採用を推奨
オプション(2)の直近不在ボーナスは、検討の結果デメリットが上回るため**現時点では見送る**。

1. **定番の「ちらつき」誤検出**: 定番でも毎スレ必ず出るとは限らず、たまたま直近数スレで途切れて再登場するケースがある。これをボーナスで新規昇格させると、排除したい定番（FLUX等）が新規欄に復活し R2 を自ら崩す。
2. **N（不在窓）の調整負担**: 小さいと短い空白で誤発火、大きいと長期休眠しか拾えず効果薄。
3. **このシステムでは「定番だけ」に作用する（最大の問題）**: 新規トピックは辞書未登録のため既に採用済み。ボーナスが実効的に効くのは「辞書登録済み＝古い定番」のみで、**主な効果は古い定番の復活**になり上振れが乏しい。
4. **狙った再燃ケースに効かない**: HiDream は辞書登録済み＆DF0.32で既に採用。Forge Neo は「不在」でなく継続言及される成熟ツール（Forge DF0.88）で、そもそも不在条件に当てはまらない。

→ 再燃を拾いたい場合は不在ボーナスより **MATURITY_RATIO 微調整** か **辞書メンテ** の方が筋が良い。

### 4-A.2 辞書（CSV）が stale である含意（重要）
`model_checked.csv`(132語) / `tool_checked.csv`(239語) は約半年前（ChromaDB導入時）に作成され、以後メンテされていない。実態確認:

- **未登録（最近の話題）**: ideogram, seedance, fable, mythos, hotgen, plamo, z-image, animerge, unsloth, smartgallery 等。さらに **novelai（111スレ出現の大定番）も未登録**。
- **登録済み（古い定番）**: anima, illustrious, flux, wan, qwen, gemma, hidream, rouwei / forge(ForgeNeo含), comfyui, reforge 等。

含意:
- **現在うまく回るのは「辞書に有る≒古い／辞書に無い≒新しい」が偶然成立しているため**。古い定番=登録済み→抑制、新規=未登録→採用、と所望の挙動になっている。
- **時間とともに劣化する**: 今後 Ideogram/seedance/Gemma4 等が定番化しても辞書に無いまま→永遠に抑制されず→**定番フラッディングが別モデル名で再発**する。
- **既に漏れている定番がある**: novelai は単独チャンクなら抑制不可（現状は他モデル併記で巻き込み抑制できているだけ）。

対処オプション:

| 案 | 内容 | コスト | 持続性 |
|---|---|---|---|
| A. 定期的な辞書リフレッシュ（半自動） | 直近スレから高頻度の新エンティティ候補を抽出→レビュー→CSV追記 | 小〜中 | 高 |
| B. 辞書非依存のDF | コーパスから動的に語彙生成（NER/パターン抽出）してDF算出 | 大 | 高 |
| C. 監視＋手動更新 | 現状0.40で運用し、フラッディング再発の兆候が出たらCSV追記 | 極小 | 中 |

スレ更新頻度が高くないなら **C（運用・監視）→ 兆候が出たら A（追記補助）** が費用対効果良。B は現時点では過剰投資。

### 4-A.3 定番フラッディングの検出方法
オプションC（監視）を実用化するための、**辞書非依存**で実行できる具体的判定方法。

**症状の定義**: 「同一の話題（エンティティ）が、抑制されないまま複数スレ連続で新規トピック欄に出続ける」状態。本来は数スレで定番化し抑制されるべきものが、辞書未登録ゆえに毎回“新規”と誤判定されている。

#### 方法B（推奨・原因特定＋修正候補を同時に得る）: 未登録エンティティのコーパスDFスキャン
1. 全格納ドキュメント（各コレクションの documents）から、エンティティ候補語を正規表現で抽出（例: ASCII英数・CamelCase・バージョン付き語 `[A-Za-z][A-Za-z0-9.\-]{2,}`）。
2. そのうち **CSV未登録** の語について、出現スレ数比率（全履歴 or 直近窓）を算出。
3. 比率 > `MATURITY_RATIO`（例0.4）または「直近Mスレ中 t スレ以上」の未登録語を抽出。
   - これらが **de-facto 定番なのに辞書漏れ＝フラッディング源**。
4. ランキング出力すれば、そのまま `model_checked.csv` / `tool_checked.csv` への**追記候補リスト**になる（オプションAの半自動メンテに直結）。

#### 方法A（軽量・継続モニタ）: 新規トピック出力の再出現監視
1. 各スレの「新規トピック」digest を保存。
2. digest から候補語（上記正規表現）を抽出。
3. 直近Kスレの digest 群で各語の出現スレ数を数え、**同一語が K スレ中 t 回以上「新規」として出続けていれば警告**。
   - 「新規欄の常連」＝抑制漏れの定番、というロジック。辞書も埋め込みも不要で最も安価。

#### 方法C（簡易・補助）: digest 自己類似度モニタ
- 今スレ digest と直近スレ digest の埋め込み類似度を計算し、**連続して高類似なら新規欄が代わり映えしていない**＝フラッディング/フィルタ不全のサイン。

#### 補助指標: 抑制率トレンド
- 抑制チャンク割合（除外/総数）が時系列で低下傾向 → 辞書劣化の間接指標。

**推奨**: 原因特定と修正が同時にできる **方法B** を月次等で回し、出力候補をCSVに追記する運用が最も実務的。日常監視を軽くしたい場合は **方法A** を併用。
