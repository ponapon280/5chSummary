**抽出結果（ツール関連のみ）**

- **ComfyUI**  
  - 複数回登場。主な話題は**マルチGPU対応の進捗**（特にROCm環境でのWindows対応改善）。  
    → 選ばれている理由：2枚挿し（例: 9070XTなど）による生成速度向上とコスパ。SUPERシリーズ待ちの文脈で「今のうちにComfyUIのマルチGPUが使えるようになったから」という実用的な選択肢として挙げられている。  
  - **Schedule Promptノード**を使った絵師タグ/画風の混ぜ方。  
    → 選ばれている理由：Anima環境で「リアスほど自由自在ではないが、現時点で一番マシな混ぜ方ができる」ため。  
  - comfyui-mcpの言及（指示ベースの自動化）。  
  - ワークフロー全体の制御性（ノードベース）。

- **sd-scripts**  
  - LoRA作成時の環境構築で複数言及。  
    → 選ばれている理由：細かい制御が可能だが、初回はPyTorchバージョン問題（50XX系でエラー）が出やすく、トラブルシューティングが必要。AIに環境構築を任せられるようになったことでハードルが下がったという文脈。

- **AI-Toolkit**  
  - LoRA作成のオールインワンツールとして登場。  
    → 選ばれている理由：「細かいことわからなくても全部やってくれる」ため、sd-scriptsより心理的ハードルが低い。ただしAnima（二次元）対応が弱い点がネックとして指摘されている。

- **Forge / A1111 (webUI系)**  
  - LoRAの記述方法（`<LoRA:xxx:1.0>`）やトリガーワードの扱いで言及。  
    → ComfyUIと比較して「Forge/A1111の方がLoRA指定を明示的に書いた方が安定する」という指摘あり。

- **その他**  
  - 特筆すべきツールの言及は上記のみ。Krea2やRedCraftなどはモデル/ファインチューン寄りの話題のため除外。

**補足**  
- モデル名（Anima、リアス、Krea2、Wan、LTXなど）は一切抽出していません。  
- Qwenシリーズの画像生成以外の話題は該当なし。  
- ツールが選ばれる理由は、主に「混ぜやすさ」「マルチGPU/速度」「環境構築の容易さ」「ノード/ワークフローの柔軟性」の観点で抽出しています。
