# 元の本文
**生成AI関連ツール レポート（変化・新登場中心）**

### ComfyUIの進化と運用変化
ComfyUIは複数回にわたり継続的に言及されているが、**バージョンアップによる挙動変化**と**運用手法の高度化**が顕著に進んでいる点が特徴的です。

- **v0.22.0→v0.23.0での変化**  
  高shift・低denoise時のi2i挙動が正常化し、denoise調整が効きやすくなったとの報告あり。Python 3.10→3.13への更新でTensorRTアップスケールの処理時間が大幅短縮（30秒→2秒）した事例も確認されています。

- **MultiGPU対応の強化**  
  公式のMultiGPU CFG Splitノードが注目されており、並列処理による速度向上が期待されています。

- **LM Studioとの連携強化（新運用）**  
  ComfyUIのカスタムノードからLM Studioを呼び出し、プロンプト生成後にモデルをunloadするワークフローが登場。VRAM24G環境でもQwen系大規模モデルと画像生成モデルの併用を可能にする実用的な手法として評価されています。この「生成完了後のunload」機能は、従来の常駐運用から大きく変化したポイントです。

- **ノード活用の洗練**  
  rgthree系ノード（Context Big、Fast Groups Muter等）の多用や、LoRA Block filterを2個使った階層別適用、NegPip・Schedule promptノードの導入が目立ちます。これらはturboモード時のネガティブプロンプト制御やワークフロー整理を目的としたもので、柔軟性が選ばれる理由となっています。

### LM Studioの位置づけ変化
従来は単独のローカルLLM/VLM実行環境として言及されていましたが、**ComfyUIとの連携ツール**としての役割が強まっています。直接呼び出し＋unload運用により、VRAM管理の観点で実用性が高まった点が新しい傾向です。一方で、ollamaからの移行先として安定して評価されており、Qwenシリーズの読み込みやすさも継続的に支持されています。

### 新たに登場・変化したツール
- **Forge Neo**  
  Forgeからの移行後にアップスケーラー（特にScuNET）の互換性問題が報告されており、プリセット切替拡張との組み合わせで運用を模索する動きが見られます。WebUI系からの移行先として実用的な選択肢の一つとなっています。

- **Krita + ComfyUI連携**  
  今回の抽出で新たに強調された連携手法。Kritaプラグイン経由でComfyUIワークフローを呼び出し、AnyTest等の追従性を保ちつつ生成結果をレイヤーとして返す運用が複数報告されています。WebUIより「描き込み」や「絵柄の一貫性」で有利という声が多く、ComfyUIの強みを活かした新しいワークフローとして注目されています。

- **Irodori-TTS Voice-Design v3**  
  v2からの機能追加（参照音声対応、テキスト＋キャプションテキスト対応）により実用性が向上したと評価されています。

- **Animarge / LECO・iLECO**  
  LoRA学習ツールとして新たに言及。Animargeは階層カット学習の簡易さと自然言語キャプション対応が選ばれる理由となっています。

- **llama.cpp（直接利用）**  
  LM Studioやollamaからの移行検討材料として登場。パフォーマンスとVRAM管理の観点で優位とされ、特にKijai版forkとの組み合わせが指摘されています。

### まとめ
前回・前々回と比較して、**ComfyUIのバージョン変化と外部ツール連携**、**Kritaとの統合**、**Forge Neoへの移行**が主な変化点です。基本的なツール（ComfyUI、LM Studio、kohya_ss）自体の存在は安定していますが、運用方法や組み合わせ方が進化しており、特にVRAM管理とワークフロー整理を目的としたカスタムノード活用が目立っています。
